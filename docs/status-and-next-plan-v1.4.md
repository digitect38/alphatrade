## 2. 프로용 자동매매에 정말 필요한 것 — 솔직한 Gap 분석

아래는 "실제 돈으로 자동매매를 돌려도 되는가?"에 대한 냉정한 평가입니다.

### 2.1 지금 있는 것 vs 프로에 필요한 것

```
구분        현재 AlphaTrade          프로 자동매매 시스템
────────    ──────────────────────   ──────────────────────────
전략 검증   백테스트 1종목/1전략       Walk-forward, Monte Carlo,
            in-sample 전체 사용        Out-of-sample 분리, 다종목 포트폴리오
                                      Regime 감지, 전략 자동 선택

실행 품질   즉시 체결 가정             체결 확인 루프, 미체결 관리,
            슬리피지 고정 0.05%        실제 슬리피지 측정/보고,
                                      VWAP/TWAP 집행 알고리즘

리스크      9개 사전 체크              사전 + 사후 체크, 실시간 P&L 모니터,
            일간 손실 킬스위치          VaR/CVaR 계산, Stress Test,

                                      불일치 시 자동 알림 + 거래 중단

복구        인플라이트 주문 복구        전체 시스템 상태 복구,
                                      DB 백업/복원 테스트,
                                      장애 시나리오별 런북
```

### 2.2 핵심 Gap 5가지

**Gap 1: 전략 신뢰도 검증 부재**
현재 백테스트는 전체 데이터에서 시그널을 생성하고 같은 데이터로 테스트합니다. 이는 과적합(overfitting) 위험이 높습니다. 실전에서는 훈련 기간과 테스트 기간을 분리하는 Walk-forward 검증이 필수이며, 최소 3~5년의 Out-of-sample 기간이 필요합니다.

**Gap 2: 체결 후 정합성 검증 없음**
주문이 FILLED로 전환된 후 실제로 브로커 계좌에 반영되었는지 확인하는 메커니즘이 없습니다. 하루가 끝나면 엔진의 포지션/현금과 브로커 실제 잔고를 대조해야 하며, 불일치 시 자동으로 거래를 중단해야 합니다.

**Gap 3: 실시간 P&L 추적 미흡**
현재 P&L은 매 사이클(~분 단위)에서만 계산됩니다. 실시간 틱 기반 P&L 계산과, 포트폴리오 전체의 실시간 손익 모니터링이 필요합니다. 특히 다수 종목 보유 시 포트폴리오 VaR 계산이 중요합니다.

**Gap 4: 장애 복구 절차 없음**
"DB가 죽으면?", "Redis가 죽으면?", "브로커 API 장기 장애면?"에 대한 문서화된 절차가 없습니다. 프로 시스템에서는 각 장애 시나리오별 대응 매뉴얼(런북)이 필수입니다.

**Gap 5: 실전 전환 검증 부족**
모의투자(VTS)에서 실전으로 전환할 때의 차이점(호가단위, 수수료율, 체결속도, API 제한)이 테스트되지 않았습니다. 실전 API는 모의투자보다 엄격한 제약이 있으며, 이 차이를 사전에 파악해야 합니다.

## 3. v1.4 구현 계획 — Production-Grade Upgrade

### 3.0 설계 원칙

1. **검증 먼저, 실행 나중**: 전략이 검증되지 않으면 실전 투입 불가
2. **정합성이 생명**: 엔진과 브로커의 상태가 항상 일치해야 함
3. **장애는 반드시 온다**: 모든 외부 의존성은 장애를 전제로 설계
4. **점진적 실전 전환**: 모의→소액 실전→정상 운용 3단계

---

### Phase 1: 전략 신뢰도 강화 (Strategy Validation)

> 목표: "이 전략이 실전에서도 통할 것인가?"에 대한 정량적 답

#### 1.1 Walk-Forward 백테스트
- 훈련:테스트 = 252일:63일 (1년:3개월) 롤링 윈도우
- 각 윈도우에서 최적 파라미터 탐색 → 다음 윈도우에서 Out-of-sample 테스트
- 모든 윈도우의 OOS 성과를 합산하여 전략 평가
- **파일**: `core-engine/app/strategy/walk_forward.py`

#### 1.2 멀티 종목 포트폴리오 백테스트
- 현재: 1종목씩만 백테스트 가능
- 개선: 유니버스 전체에 대한 동시 시뮬레이션
- 포지션 간 상관관계, 섹터 집중도 제약 반영
- **파일**: `core-engine/app/strategy/portfolio_backtest.py`

#### 1.3 전략 성과 리포트
- Sharpe, Sortino, Calmar 비율
- 월별/분기별 수익률 히트맵
- 최대 연속 손실 기간, Recovery Factor
- 벤치마크(KOSPI) 대비 Information Ratio
- **UI**: 백테스트 페이지에 상세 결과 탭 추가

#### 1.4 전략 자동 선택 (Regime Detection)
- 시장 변동성 레짐 감지 (Low/Medium/High volatility)
- 레짐별 최적 전략 프리셋 자동 전환
- VIX에 해당하는 VKOSPI 활용
- **파일**: `core-engine/app/strategy/regime.py`

### Phase 2: 실행 품질 고도화 (Execution Quality)

> 목표: 주문이 의도대로 체결되었는지 100% 확인 가능

#### 2.1 체결 확인 루프
- ACKED 상태 주문에 대해 5초 간격으로 체결 상태 폴링
- 미체결 1분 초과 시 알림, 5분 초과 시 자동 취소 검토
- KIS API `TTTC8001R` (주문 체결 조회) 활용
- **파일**: `core-engine/app/execution/fill_monitor.py`

#### 2.2 실제 슬리피지 측정
- 시그널 발생 가격 vs 실제 체결 가격 차이 기록
- 종목별/시간대별 슬리피지 통계
- 슬리피지가 임계값 초과 시 알림
- **DB 테이블**: `execution_quality` (time, order_id, signal_price, fill_price, slippage_bps)

#### 2.3 EOD 정합성 검증 (Reconciliation)
- 매일 15:40 (장 종료 후) 자동 실행
- 3중 대조: 엔진 포지션 / 브로커 잔고 / DB 주문 내역
- 불일치 항목 자동 보고 + 킬스위치 발동 옵션
- **파일**: `core-engine/app/execution/reconciliation.py`
- **API**: `POST /trading/reconcile`

#### 2.4 미체결 주문 관리
- 장 종료 시 잔존 미체결 주문 자동 취소
- 부분 체결(PARTIALLY_FILLED) 잔량 처리 정책
- 주문 유효기간(TTL) 설정
- **파일**: `core-engine/app/execution/order_cleanup.py`

### Phase 3: 리스크 관리 고도화 (Advanced Risk)

> 목표: "최악의 상황에서도 자본을 보존할 수 있는가?"

#### 3.1 실시간 포트폴리오 P&L
- WebSocket 틱 기반 실시간 손익 계산
- 포트폴리오 전체 합산 P&L 대시보드
- 분 단위 P&L 히스토리 저장
- **파일**: `core-engine/app/risk/realtime_pnl.py`

#### 3.2 포트폴리오 VaR/CVaR
- Historical VaR (95%, 99% 신뢰구간)
- Conditional VaR (Expected Shortfall)
- 포지션별 기여 VaR (Marginal VaR)
- 일간 보고 + 한도 초과 시 알림
- **파일**: `core-engine/app/risk/var_calculator.py`

#### 3.3 Stress Test
- 시나리오 기반: 2020 코로나 폭락, 2022 금리 폭등, 서킷브레이커 발동
- 현재 포트폴리오에 과거 극단 시나리오 적용
- 최대 손실 예상액 보고
- **파일**: `core-engine/app/risk/stress_test.py`

#### 3.4 상관관계 기반 집중 리스크
- 기존 상관관계 분석을 리스크에 통합
- 상관계수 > 0.7인 종목 그룹은 단일 포지션으로 취급
- 유효 분산(Effective Diversification) 점수
- **파일**: `core-engine/app/risk/concentration.py`

### Phase 4: 운영 안정성 (Operational Resilience)

> 목표: "시스템 장애에서도 안전하게 복구할 수 있는가?"

#### 4.1 자동 백업/복구
- TimescaleDB: 일간 pg_dump + S3/로컬 저장 (7일 보관)
- Redis: RDB 스냅샷 + AOF (orders, kill_switch 등 핵심 키)
- 복구 드릴: 월 1회 백업으로부터 복원 테스트
- **파일**: `scripts/backup.sh`, `scripts/restore.sh`

#### 4.2 인시던트 런북
- 시나리오 1: 브로커 API 장애 (30분 이상 응답 없음)
- 시나리오 2: DB 연결 장애
- 시나리오 3: 중복 주문 의심
- 시나리오 4: 킬 스위치 발동 후 복구
- 시나리오 5: 장 중 시스템 재시작
- **파일**: `docs/runbook-incidents.md`

#### 4.3 알림 에스컬레이션
- Level 1 (정보): 텔레그램만 (체결, 시그널)
- Level 2 (경고): 텔레그램 + 이메일 (손실 경고, 시스템 지연)
- Level 3 (위험): 텔레그램 + 이메일 + 반복 알림 (킬스위치, 정합성 불일치)
- 알림 속도 제한: 동일 이벤트 5분 쿨다운
- **파일**: `core-engine/app/services/alert_escalation.py`

#### 4.4 시크릿 관리 개선
- `.env` 파일에서 Docker secrets 또는 환경변수 주입으로 전환
- KIS API 키, DB 비밀번호 등 민감 정보 분리
- 키 로테이션 절차 문서화
- **파일**: `docs/secrets-management.md`

### Phase 5: 실전 전환 (Paper → Live Migration)

> 목표: "모의투자에서 실전으로 안전하게 전환"

#### 5.1 실전 전환 체크리스트
- [ ] Walk-forward 백테스트 통과 (OOS Sharpe > 0.5)
- [ ] 모의투자 2주 연속 운영, 정합성 불일치 0건
- [ ] EOD Reconciliation 14일 연속 통과
- [ ] 킬 스위치 동작 테스트 완료
- [ ] 백업/복원 드릴 1회 완료
- [ ] 런북 검토 완료
- [ ] 실전 API 키 발급 및 테스트 (소액 1주 매매)

#### 5.2 KIS 실전 API 전환
- 모의투자 URL → 실전 URL 전환 (설정 변수)
- TR_ID 전환 (VTTC* → TTTC*)
- 호가단위 차이 반영 (모의: 없음 → 실전: 종목별 틱 사이즈)
- 수수료/세금 실전 값 적용
- **파일**: `core-engine/app/config.py` (KIS_MODE: paper|live)

#### 5.3 점진적 자본 투입
- Stage 1: 총 자본 100만원, 종목당 최대 20만원 (2주)
- Stage 2: 총 자본 500만원, 종목당 최대 50만원 (4주)
- Stage 3: 정상 운용 (설정된 한도 적용)
- 각 Stage 종료 시 성과 리뷰 + 승인 게이트

#### 5.4 실전 모니터링 강화
- 체결률 (주문 대비 체결) 추적
- 체결 지연시간 측정
- 일간 P&L 실시간 SMS 알림 (손실 -1% 초과 시)
- 주간 성과 자동 리포트 (이메일)

## 5. 성공 기준

### 모의투자 졸업 조건
- Walk-Forward OOS Sharpe Ratio > 0.5
- 14일 연속 EOD Reconciliation 통과 (불일치 0건)
- 킬 스위치/런북 테스트 완료
- 백업 복원 드릴 1회 성공

### 실전 운용 기준
- 월간 MDD < -5%
- 일간 최대 손실 < -2% (킬 스위치 자동)
- 체결률 > 95%
- 평균 슬리피지 < 10bps
- 시스템 가용률 > 99.5% (장중)
