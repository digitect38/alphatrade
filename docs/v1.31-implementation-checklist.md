# v1.31 Implementation Checklist

Source plan:
- `plan/AlphaTrade_Development_Plan_v1.31.md`

Purpose:
- Convert the `v1.31` plan into an execution checklist with concrete done criteria.

## 1. Phase A: Minimum Live Trading Control Package

These items must be complete before any real-money pilot.

### A-1. Risk Policy

- [ ] Define account-level daily loss limit in config and docs
- [ ] Define strategy-level daily loss limit in config and docs
- [ ] Define symbol max allocation rule
- [ ] Define sector concentration rule
- [ ] Define liquidity participation rule
- [ ] Define session-entry windows and forbidden periods
- [ ] Define stale-price threshold and block behavior
- [ ] Add automated enforcement in order pre-checks
- [ ] Add tests for every rejection path

Done when:
- Risk rules are documented, configured, enforced in code, and covered by tests.

### A-2. Order FSM And Idempotency

- [ ] Implement explicit order states: `created`, `validated`, `submitted`, `acked`, `partially_filled`, `filled`, `cancelled`, `rejected`, `expired`, `unknown`
- [ ] Add idempotency key generation and persistence
- [ ] Prevent duplicate submission on retries and restart
- [ ] Add broker ACK timeout handling
- [ ] Add restart recovery for in-flight orders
- [ ] Add tests for partial fill, reject, timeout, duplicate submit

Done when:
- An order can be recovered deterministically after restart without duplicate execution.

### A-3. Broker Reconciliation

- [ ] Add end-of-day position reconciliation
- [ ] Add end-of-day cash reconciliation
- [ ] Add open-order reconciliation
- [ ] Persist reconciliation results and mismatch reasons
- [ ] Add alert on mismatch severity thresholds

Done when:
- Internal state and broker state are compared automatically every day and mismatches are visible.

### A-4. Bad Data Gate

- [ ] Validate stale prices
- [ ] Validate missing prices
- [ ] Validate outlier prices
- [ ] Validate missing symbol metadata
- [ ] Block new orders when market data quality checks fail
- [ ] Alert operators on data-gate activation

Done when:
- Trading entry is blocked automatically whenever input quality is not trustworthy.

### A-5. Kill Switch

- [ ] Add automatic kill switch for daily loss breach
- [ ] Add automatic kill switch for repeated broker failures
- [ ] Add manual kill switch API
- [ ] Add manual kill switch dashboard control
- [ ] Persist kill switch reason and actor
- [ ] Add unlock workflow with audit trail

Done when:
- New entries can be stopped immediately by policy or operator, and the reason is recorded.

### A-6. Immutable Audit Log

- [ ] Create append-only audit table/schema
- [ ] Log strategy decisions
- [ ] Log risk decisions
- [ ] Log order requests
- [ ] Log broker responses
- [ ] Log manual operator actions
- [ ] Add correlation IDs across the full flow
- [ ] Define archive and retention process

Done when:
- Every trading action can be reconstructed chronologically from stored events.

### A-7. Version Pinning

- [ ] Pin Docker image versions in compose
- [ ] Pin Python dependency versions
- [ ] Pin Node dependency versions
- [ ] Pin LLM model names and versions where applicable
- [ ] Document upgrade procedure through staging

Done when:
- Production deployment is reproducible and does not rely on floating versions.

### A-8. VPS-First Live Execution

- [ ] Separate research/monitoring node from execution node
- [ ] Move broker execution path to VPS
- [ ] Keep MacBook out of direct live order execution
- [ ] Validate degraded mode when analysis node is offline
- [ ] Validate protection mode on execution node

Done when:
- Live orders can continue to be protected even if the local workstation is unavailable.

## 2. Phase B: Operational Hardening

### B-1. Backup And Recovery

- [ ] Define RPO and RTO in ops docs
- [ ] Add scheduled incremental DB backup
- [ ] Add daily full snapshot
- [ ] Add monthly restore drill
- [ ] Record restore results and failures

Done when:
- Backups are automated and restore success is proven periodically.

### B-2. Durable Event Storage

- [ ] Persist critical order and position events in DB or WAL-backed storage
- [ ] Keep Redis only as delivery path, not sole record
- [ ] Add replay mechanism for recovery

Done when:
- Critical trading events survive restart and can be replayed.

### B-3. Incident Runbooks

- [ ] Write broker outage runbook
- [ ] Write market data outage runbook
- [ ] Write duplicate-order incident runbook
- [ ] Write restart recovery runbook
- [ ] Define alert severities and operator response windows

Done when:
- Operators have written actions for the main failure modes.

### B-4. Security Operations

- [ ] Define RBAC roles and boundaries
- [ ] Add secret rotation schedule
- [ ] Add image vulnerability scanning
- [ ] Add dependency vulnerability scanning
- [ ] Add webhook signature validation

Done when:
- Access, secrets, and software supply chain controls are documented and enforced.

### B-5. Release Controls

- [ ] Define staging-to-production promotion flow
- [ ] Require approval before production release
- [ ] Tag releases with versioned changelog
- [ ] Record deployment operator and timestamp

Done when:
- Production changes are intentional, reviewed, and traceable.

### C-1. Promotion Stages

- [ ] Define minimum duration per stage
- [ ] Define pass/fail thresholds per stage

Done when:
- Strategy promotion is rule-based instead of discretionary.

### C-2. Walk-Forward And Regime Testing

- [ ] Add walk-forward validation workflow
- [ ] Add regime segmentation tests
- [ ] Record performance by regime

Done when:
- Strategy quality is evaluated outside a single historical fit.

### C-3. Execution Quality Metrics

- [ ] Track slippage
- [ ] Track turnover
- [ ] Track order latency
- [ ] Track fill ratio
- [ ] Show metrics in Grafana or dashboard

Done when:
- Live execution quality is visible and measurable.

### C-4. LLM Governance

- [ ] Version prompts
- [ ] Version model choices

Done when:
- LLM usage is controlled, auditable, and bounded in cost.

Do not start live trading until all are true:

- [ ] Phase A is complete
- [ ] Restore drill has passed at least once
- [ ] Broker reconciliation has passed for multiple sessions
- [ ] Kill switch has been tested in staging
- [ ] Duplicate-order prevention has been tested
- [ ] Audit log is queryable end-to-end
- [ ] Live execution runs on VPS, not on MacBook
